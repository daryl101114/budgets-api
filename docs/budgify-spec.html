1. Overview

Budgify is a personal budgeting REST API that allows users to track income and expenses across multiple wallets, categorize transactions, plan budgets, and import transactions from CSV bank exports.

Base URL: http://localhost:{port}/api

The original design doc used versioned URLs (/api/v1/). The implemented API uses unversioned routes (/api/).

4. Authentication

JWT tokens are issued on login and delivered as an app.at HttpOnly cookie. Clients must extract the token from the cookie and send it as a Bearer token on subsequent requests:

Authorization: Bearer <token>

All endpoints except POST /api/authentication/register and POST /api/authentication/login require a valid token.

The original design doc used the path /api/auth/. The implementation uses /api/authentication/.

6.8 Import Batches — /api/import

Requires authentication. Import is a multi-step flow:

Step 1: Upload CSV        →  batch created  (Status: Reviewing)
Step 2: Review            →  batch approve / reject duplicates
Step 3: Confirm or Rollback

Step 1 — Upload CSV

POST/api/import/wallets/{walletId}

Content-Type: multipart/form-data
Form field: file — must be a .csv file

Expected CSV columns:

ColumnRequiredNotes
DateYesParseable date string
AmountYesDecimal
TypeYesIncome or Expense
DescriptionNoUser-facing description
RawDescriptionNoOriginal bank description
BankTxnTypeNoBank-provided type label
CheckNumberNo
BalanceAfterNoDecimal — running balance from bank

Duplicate detection: For each row, a SHA256 hash is computed and compared against all non-deleted transactions in the wallet:

input  = "{walletId}|{date:yyyy-MM-dd}|{amount}|{rawDescription?.ToLower().Trim()}"
hash   = SHA256(UTF8(input)).ToHexString().ToLower()   // 64-char hex

If the hash already exists, IsDuplicate = true is set on the imported transaction (it is NOT automatically skipped).

Response: 201 Created — ImportBatchDto

{
  "id": "guid",
  "walletId": "guid",
  "fileName": "export.csv",
  "status": "Reviewing",
  "totalRows": 10,
  "importedCount": 0,
  "duplicateCount": 3,
  "skippedCount": 0,
  "createdAt": "2026-01-01T00:00:00Z",
  "confirmedAt": null,
  "rolledBackAt": null
}

Get All Batches

GET/api/import

Returns list of ImportBatchDto for the authenticated user.

Step 2 — Review Batch

GET/api/import/{id}

Returns ImportBatchDetailDto (extends ImportBatchDto):

{
  ...ImportBatchDto fields...,
  "pendingDuplicates": 3,
  "readyToConfirm": false,
  "transactions": [
    {
      "id": "guid",
      "amount": 100.00,
      "transactionType": "Income",
      "date": "2026-01-15",
      "description": "Payroll",
      "rawDescription": "PAYROLL DEPOSIT",
      "bankTxnType": "Credit",
      "isDuplicate": true,
      "importStatus": "Pending"
    }
  ]
}
FieldDescription
pendingDuplicatesCount of transactions where IsDuplicate=true AND ImportStatus=Pending. Computed in service layer — not stored.
readyToConfirmpendingDuplicates == 0. Informational — confirm is NOT blocked by this flag.

Step 2b — Batch Approve Duplicates

PATCH/api/import/{id}/transactions/approve

Sets ImportStatus = Approved on specified transactions.

{
  "transactionIds": ["guid", "guid"]
}

Step 2c — Batch Reject Duplicates

PATCH/api/import/{id}/transactions/reject

Sets ImportStatus = Rejected on specified transactions.

{
  "transactionIds": ["guid", "guid"]
}

Step 3a — Confirm Batch

POST/api/import/{id}/confirm

Finalizes the batch:

  1. Transactions with ImportStatus = Rejected → soft-deleted, counted in SkippedCount
  2. Remaining transactions (Pending + Approved) → ImportStatus = Confirmed, counted in ImportedCount
  3. Wallet balance adjusted per transaction type (Income: +amount, Expense: −amount)
  4. Batch.Status = Confirmed, ConfirmedAt = UTC now

Response: 200 OK — updated ImportBatchDto
Error: 400 if batch status is not Reviewing

Step 3b — Rollback Batch

POST/api/import/{id}/rollback

Reverses a confirmed batch using delta reversal (not full recalculation):

  1. Loads all non-soft-deleted batch transactions (these are all Confirmed at this point)
  2. Reverses wallet balance delta: Income → balance -= amount, Expense → balance += amount
  3. Soft-deletes all non-soft-deleted batch transactions
  4. Batch.Status = RolledBack, RolledBackAt = UTC now

Response: 200 OK — updated ImportBatchDto
Error: 400 if batch status is not Confirmed

A Reviewing batch cannot be rolled back — abandon or delete it instead.

ImportBatchStatus change: The original design doc defined 5 states: PENDING, REVIEWING, COMPLETED, ROLLED_BACK, FAILED. The implementation simplifies to 3 states: Reviewing, Confirmed, RolledBack. There is no explicit FAILED state — errors are surfaced as HTTP 400/500 responses.

All enums that are stored in the database use string conversion (not integer values) for readability and migration safety.

8. Design Patterns

AutoMapper PreCondition for Nullable Value Types

For nullable value type fields in Update DTOs, use PreCondition (not Condition) to skip the mapping when the source is null:

.ForMember(dest => dest.Amount, opt => opt.PreCondition(src => src.Amount.HasValue))

Never Call .Update() on EF Core Tracked Entities

After a .FindAsync() or .FirstOrDefaultAsync(), the entity is already change-tracked. Modify properties directly and call SaveChangesAsync() — calling .Update() is redundant and can cause issues.

Bulk Updates via ExecuteUpdateAsync

Use ExecuteUpdateAsync when updating many rows without needing to load entities into memory (e.g., batch approve/reject).

Soft Deletes

All main entities support soft delete via a DeletedAt timestamp. All queries filter DeletedAt == null by default.

FK Cascade Cycle Fix (SQL Server Error 1785)

SQL Server rejects migrations where multiple cascade paths exist between two tables. Fix: use DeleteBehavior.Restrict on the secondary FK path. Applied to ImportBatch → User (UserId).

modelBuilder.Entity<ImportBatch>()
    .HasOne(b => b.User)
    .WithMany()
    .HasForeignKey(b => b.UserId)
    .OnDelete(DeleteBehavior.Restrict);

Shared DbContext per Request

Repositories are registered as Scoped, sharing one DbContext instance within a request. Changes made by one repository are immediately visible to another without re-querying.

Delta-Based Balance Updates

Wallet balance is always adjusted by a delta (+amount / -amount), not recalculated from all transactions. This applies to both manual CRUD and import confirm/rollback.
